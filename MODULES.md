# HONEY BEE Operation — モジュール仕様（現状準拠）

実運用の React アプリは **`honey-bee-eventsnpm/`** です。ルート直下の `src/` は CRA 初期テンプレートであり本体ではありません。

この文書には **API キー・パスワード・Firebase 設定値などの秘密は書きません。** 機密の扱いは末尾のセクションも参照してください。

---

## アプリ構成とナビゲーション

グローバルナビ（ホーム以外）で次のモジュールに移動できます。

| 画面名 | 主なソース | 概要 |
|--------|------------|------|
| 本日の営業 | `today.js` | 当日のイベント・貸切・シフト・予約の把握、業務チェックリスト、申し送り |
| 予約管理 | `reservation.js` | 予約の登録／一覧／カレンダー、CSV取込、来店チェック、席番号など |
| イベント管理 | `App.js`（一覧・編集 UI） | イベント CRUD、カレンダー／リスト、CSV、告知文出力、テンプレ、タイムテーブル、AI説明文など |
| 貸切 | `rentals.js` | 貸切案件のワークフロー、見積／請求、返信ドラフト AI、イベントとの連携 |
| アーティスト精算 | `settlement.js` | 歩合／固定の計算、精算メモの生成・コピー |
| シフト | `shift.js` | 勤務表 CSV の取込み・月別表示 |

席レイアウト編集・日表示は **`seatLayout.js`** から予約フロー内で開きます。

### お客様向け予約フォーム

URL に **`?reserve=1`** が付いたとき、スタッフ向けとは別にお客様用の予約フォームのみを表示します（実装は `App.js` と `reservation.js`）。イベントは **`bookingStatus === open`** のみ選択可能（`eventBooking.js` の判定を利用）。

## Firestore コレクション（コード上）

アプリから参照・更新している主なコレクションです（名前のみ。フィールド詳細は各画面・アプリ側を正とします）。

| コレクション | 主な用途 |
|--------------|----------|
| `events` | イベント |
| `templates` | イベント用テンプレート |
| `rentals` | 貸切案件 |
| `shifts` | シフト（月度データ） |
| `reservations` | 予約 |
| `seatLayouts` | 座席レイアウト定義 |
| `daily` | 日単位データ（日表示のレイアウト選択、ブロック席、本日の営業のチェック状態など） |
| `handovers` | 申し送りアイテム |
| `settlements` | アーティスト精算データ |
| `counters` | 連番管理（見積番号など） |

## 実装済みモジュール（機能の整理）

### 1. イベント管理（`App.js` 中心）

- イベントの新規作成・編集・保存、論理削除とゴミ箱（一定期間後のpurge）
- 一覧：**カレンダー表示**／**リスト表示**
- **CSV 読み込み**（複数イベント名や昼／夜などのパターンへの分割ロジックを含む）。貸切っぽい行の **貸切モジュールへの自動登録**に連動する処理あり
- テンプレートの保存／読込（`templates`）
- **告知文などの自動生成**：HP／Instagram／Facebook／フォーム案内／その他チャネル、Wix 向け複数パーツ、フライヤー用テキスト、タイムテーブル用テキストなど（ルールベースの `generateTexts`）
- **OpenAI**：イベント説明文の AI 生成（利用には利用者側で API キーをアプリから保存。**キー値は環境によりブラウザに残る。**）
- タイムテーブル編集タブ（行の追加・テンプレ読込・コピー用テキスト）
- イベントと貸切の **相互変換**（フローはアプリ UI 上）
- 画像 URL の複数枚、ポスターリンク、予約関連フラグなどのフィールド
- **`bookingStatus`**（`events` ドキュメント）: `pending_detail`（詳細確認中）／`pending_booking`（予約受付前）／`open`（予約受付中）／`closed`（受付終了）。**未設定の既存データは `open` とみなす**（公開予約フォームの判定）。CSV **新規**取込および**手動新規**イベントの既定は `pending_detail`。CSV **既存行の取り込み**では `bookingStatus` を上書きしない。**`noBooking`** は従来どおり「一切予約しない」強いフラグで、`bookingStatus` より公開フォーム上は優先。

### 2. 予約管理（`reservation.js`）

- 予約の **リアルタイム一覧** と **カレンダー／日別**ビュー
- 予約項目：イベント名・日付・名前・人数・電話・メール・メモ・**予約経路**（電話／フォーム等）・**受付スタッフ**・**来店済み**・**席番号**（複数席対応）など
- **CSV インポート**、ゴミ箱と一定期間後の自動削除
- メール送信：**EmailJS** 経由の店舗向け／お客様向け通知（`'./email'`）
- 日付キー単位での **レイアウト選択**・**ブロック席**と `seatLayouts` / `daily` 連携
- 「予約一覧とイベント名の整合」などの補助 UI

### 3. 席レイアウト（`seatLayout.js`）

- レイアウト CRUD（座席配置、背景画像、`daily` と日付での紐づけ）
- 特定日での予約状況に応じた席の状態表示（空／予約／来店済／ブロックなど）

### 4. 本日の営業（`today.js`）

- 日付選択で **当日のイベント／貸切／シフト／予約のサマリー** を表示
- **業務チェックリスト**（開店前〜閉店のテンプレ項目）。チェック状態は **`daily`** に保存され、並列編集を意識したフィールド更新
- **申し送り**（`handovers`）：翌日単発・複数日・期間指定などモードあり
- イベント編集・予約画面へのジャンプなどのオペ連携

※旧版 MODULES で「チェックリスト未実装」「本日の営業未実装」とあった記述は、現コードでは **実装済み**です。

### 5. 貸切管理（`rentals.js`）

- 問い合わせ〜担当・ステータス・連絡先・要件（人数・料理・機材フラグなど）・メモ
- **ステータス**（未対応／返信済み／…／完了など：コード `RENTAL_STATUSES` に準拠）
- **見積書・請求書** の HTML 出力・連番、`documentHistory`、`counters`
- OpenAI を使った **返信文ドラフト生成**（アプリ側の API キー設定前提）
- イベント一覧からの **一括取込**、`App.js` 側のイベント変換との連携
- 予約金（デポジット）フラグなどの運用項目

※旧 MODULES の「貸切未実装」は outdated です。**実装済み**です。

### 6. アーティスト精算（`settlement.js`）

- イベント日・イベント名に紐づく **精算レコード**（一覧・フィルタ：支払済み等）
- 出演者ごとに：**入場料／動員・歩合率・最低保証・固定額**、**差引き項目の複数行**、**支払方法・支払済みフラグ・メモ** など（実際の入力項目は画面を正）
- **精算メモの自動生成とクリップボードコピー**
- 過去一覧の参照に **ブラウザ内の軽い制限（パスワード）** がある（**パスワード文字列はドキュメントに書かない。コードのみ参照。**）
- ゴミ箱と論理削除

旧 MODULES が列挙した「予約人数と当日人数の別フィールド」「源泉徴収」「交通費」などは、**仕様書レベルでは未充足のまま**の可能性があります。次章「部分実装・拡張余地」参照。

### 7. シフト管理（`shift.js`）

- 月度 **CSV 取込**とパース（セル内の役割・時間などの解釈）
- Firestore `shifts` への保存と月別表示
- 予約モジュールなどで **スタッフ名の並び**に利用されるヘルパ

### 8. メール（`email.js`）

- **EmailJS** の動的ロードと、予約データからの **店舗向け／お客様向け**送信

## 部分実装・データモデル上の拡張余地

次の項目は、過去の仕様書や現場要望と比べると **画面・フィールドが簡略化されている、または「メモ等で代替」の可能性**があります。実装の正はコードと画面です。

| 領域 | 内容 |
|------|------|
| アーティスト精算 | 「予約人数」と「当日人数」の**別フィールド**、**源泉徴収**・**交通費**などの明示項目が、旧 MODULES で求めていた粒度と完全一致しない可能性 |
| 本日の営業と精算・予約の連携 | 一画面での「すべての自動集計」ではなく、`daily` ベースのチェックと一覧表示が中心。**精算一覧の自動埋め込み**などは限定的 |

## 機密情報とセキュリティ（注意のみ・値は書かない）

- **Firebase**：クライアント SDK に渡す設定はリポジトリのソースに含まれうる。真正のゲートキーパは **Firestore Security Rules と Authentication**。**ルールを未設定／緩すぎるまま公開 URL を配ることは禁止レベルで危険**です。
- **EmailJS**：サービス ID・テンプレート ID・公開キー等はコードに含まれることが多い。漏えい時は各ダッシュボードでローテーション。
- **OpenAI API キー**：ブラウザの localStorage 等に保存される設計。共有端末・開発者コンソール・XSS リスクへの配慮が必要。
- **パスワード文字列**：精算履歴の簡易ロックなどがコードベースにある場合がある。**ドキュメントに値を転記しない。** 運用強度は「サイト全体の公開範囲・Rules・認証」で担保する前提で設計することが望ましい。
